## What runs in the per-round loop now

`tools/train/run_training.sh` was rewritten by the s0_pcfg session to
chain four binaries per training round. Each is md5-cached and silently
skipped if its source is missing:

```
self_train round            /tmp/rail_st_bin           (Empire transplant Session 1 hooks)
  ↓
recovery flush              /tmp/rail_flush_bin        (Session 2 — 5 protected files)
  ↓
s0_pcfg tick                /tmp/rail_s0d_tick_bin     (REINFORCE step, 30 samples)
  ↓
s0_pcfg cross_feed          /tmp/rail_s0d_xfeed_bin    (push verified PCFG output → harvest.jsonl)
  ↓
sleep 2 → next round
```

The whole loop closes: PCFG produces compiler-verified Rail → cross_feed
translates it into chat-completion format → self_train's harvest pipeline
picks it up → next training round trains on it.

Every event from either trainer lands in the same JSONL ledger. Every
weight write logs a diff. Every round's state survives in a guarded
backup chain. The whole observability + tuning loop is one closed system.

## ⚠️ Test runner race — important quirk

`./rail_native test` writes to `/tmp/rail_out` between phases. If TWO
processes invoke the compiler at the same time (e.g. a test run + a
parallel `run_training.sh` loop + a polling test watcher), they stomp
each other and you'll see *anywhere from 11/92 to 89/92*. **This is
not a regression, it's a /tmp/rail_out race.**

**The canonical test count is 92/92.** Both sessions confirmed it
independently in their commit messages today (`40cc5a6`, `8209651`,
`263fd8c`, `7af5fa6`, `9af7758`).

If you see anything below 92/92 after a fresh checkout:
1. `ps aux | grep rail_native` — kill any stale `rail_native test` processes
2. Make sure `run_training.sh` isn't running in another terminal
3. Re-run `./rail_native test` — it should hit 92/92

The fix would be to give the test runner an isolated `/tmp/rail_test_$$`
output path, but that's a compile.rail change deferred to a future session.
